# Project Overview

## 1. Course and Assignment Context

This project was developed for the **Agile Project Management** course as part of a group assignment focused on the application of **Agile, Scrum, DevOps and Artificial Intelligence** concepts in a realistic product context.

The assignment proposes the creation of an **AI DevOps Assistant for Software Teams**. The solution should support software teams in activities related to planning, execution, quality, delivery and continuous improvement.

The project uses **Scrum** as the management framework and **DevOps** as the delivery and improvement approach. The objective is not only to create a functional prototype, but also to demonstrate how Agile Project Management concepts can be applied in practice.

---

## 2. Project Purpose

**AI Risk Detector** is an AI-supported assistant designed to help Agile and DevOps teams analyse software tasks, requirements or backlog items before execution.

The purpose of the product is to help teams identify possible risks earlier, improve backlog quality and support better planning and release decisions.

Instead of replacing human decision-making, the assistant provides structured recommendations that can be reviewed by the Product Owner, Scrum Master, Developers or other relevant stakeholders.

The project therefore combines two perspectives:

- a **product perspective**, where AI Risk Detector acts as a lightweight assistant for software teams;
- a **management perspective**, where the project demonstrates the use of Scrum, backlog management, DevOps reasoning and continuous improvement.

---

## 3. Main Problem

Agile software teams often need to deliver value quickly while dealing with uncertainty, changing requirements and technical complexity.

However, fast delivery can create problems when work items are not sufficiently clear or well prepared. Common issues include:

- poorly structured backlog items;
- ambiguous requirements;
- missing or weak acceptance criteria;
- limited visibility of delivery and technical risks;
- lack of clear testing expectations;
- release readiness uncertainty;
- rework caused by late identification of problems;
- weak connection between planning, development, testing and delivery.

These problems can affect sprint execution, product quality, team confidence and delivery predictability.

For example, a team may start implementing a user story that is too large, unclear or missing acceptance criteria. Another team may prepare a deployment without rollback planning, monitoring or sufficient test coverage. In both cases, the risk is discovered too late, increasing the probability of delays, incidents or rework.

AI Risk Detector aims to support earlier discussion of these risks.

---

## 4. Product Summary

AI Risk Detector receives a software task, requirement or user story written in natural language and returns a structured risk analysis.

The expected output may include:

- **Risk level** — a simple classification such as Low, Medium or High;
- **Potential risks** — possible issues that may affect delivery, quality or stability;
- **Estimated impact** — explanation of what may happen if the risk is not managed;
- **Mitigation recommendations** — actions that can reduce the identified risks;
- **Suggested tests** — possible checks or test scenarios to validate the work item;
- **Possible acceptance criteria** — criteria that can help clarify when the item is complete;
- **Improvement suggestions** — recommendations to make the backlog item clearer or more actionable.

The assistant can be used during:

- backlog refinement;
- Sprint Planning;
- technical discussion;
- release preparation;
- Sprint Review;
- Sprint Retrospective.

The output is intended to support conversation and decision-making inside the team. It should always be validated by humans before being used in real project decisions.

## 6. Scope Clarification

The MVP is intentionally lightweight.

This project does not aim to build a complete enterprise AI platform, a production-ready DevOps tool or a fully automated release management system.

The main objective is to demonstrate that the group can apply Agile Project Management concepts in a coherent way, including:

- defining a product vision;
- structuring and prioritising a backlog;
- writing user stories and acceptance criteria;
- applying Scrum roles, artifacts and events;
- documenting DevOps practices and limitations;
- demonstrating a functional MVP concept;
- reflecting critically on the use of AI in Agile and DevOps contexts.

The application is therefore used as evidence of the project idea, while the main academic value comes from the management approach, documentation, Scrum evidence, DevOps reasoning and critical reflection.

---

## 7. Project Logic

The project follows this logic:

> Problem → Product Vision → Product Backlog → Scrum Execution → DevOps Evidence → MVP Demonstration → Critical Reflection

This structure ensures that the project is not only presented as a software prototype, but as a complete Agile Project Management case.

Each repository document contributes to this logic and supports a specific part of the final evaluation.
