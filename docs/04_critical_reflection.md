# Critical Reflection

## Purpose

This document contains the final reflection on the use of Agile, Scrum, DevOps, and AI in the AI Risk Detector project.

## Reflection on Agile and Scrum

The project applies Agile thinking by focusing on a small MVP, iterative improvement, backlog prioritization, and feedback.

Scrum helps structure the work through roles, artifacts, events, sprint planning, review, and retrospective reflection.

One limitation is that Scrum documentation can become fragmented if every event is documented in a separate file. For this reason, detailed operational evidence is kept in Jira and the repository documentation has been simplified.

## Reflection on DevOps

DevOps is represented both in the project delivery approach and in the product idea.

The delivery approach uses GitHub, source control, dependencies, tests, and workflow evidence. The product idea supports DevOps teams by identifying delivery and release risks earlier.

One limitation is that the MVP does not include full deployment automation, monitoring, or production integration. These are future improvements.

## Reflection on AI Use

AI can help teams identify risks, suggest tests, improve acceptance criteria, and make planning conversations more structured.

However, AI output can be incomplete, generic, or incorrect. The quality of the output depends on the input and on the context available to the system.

Human validation is required before using AI recommendations for real project decisions.

## Key Limitations

- The MVP is lightweight and academic.
- The AI output needs human review.
- Jira integration is represented conceptually and through Jira evidence, not through a full API integration.
- The product does not replace Scrum roles or team decision-making.

## Future Improvements

Possible future improvements include:

- Jira export or API integration
- Stronger prompt validation
- More automated tests
- Risk history
- Release readiness scoring
- Monitoring or incident feedback
- Better evidence dashboards
