# Product Backlog and Scrum

## Purpose

This document summarizes how the Product Backlog and Scrum approach are used in the AI Risk Detector project.

The detailed operational evidence is maintained in Jira and will be presented online.

## Product Backlog

The Product Backlog contains epics, user stories, acceptance criteria, priorities, story points, sprint allocation, and workflow status.

The Jira board is the main source for:

- Epics
- Product Backlog
- User Stories
- Acceptance Criteria
- Priority Score 1-4
- Story Points
- Sprint allocation
- Workflow: To Do -> In Progress -> In Review -> Done

## MVP Backlog Focus

The MVP backlog focuses on the core user flow:

1. The user enters a task, requirement, or user story.
2. The assistant analyses the input.
3. The assistant returns a structured risk report.
4. The team reviews the output and uses it to improve planning decisions.

## Scrum Roles

The project applies Scrum roles in an academic context:

- Product Owner perspective: defines value, prioritizes the backlog, and clarifies the MVP.
- Scrum Master perspective: supports the Scrum process, transparency, and continuous improvement.
- Developers perspective: creates the increment, documentation, and project evidence.

## Scrum Events

The project uses the Scrum event logic:

- Sprint Planning: select work and define the sprint goal.
- Daily coordination: keep progress visible and identify blockers.
- Sprint Review: inspect the increment and collect feedback.
- Sprint Retrospective: reflect on the process and define improvement actions.

## Definition of Ready

A backlog item is ready when it has:

- Clear user value
- A user story or task description
- Acceptance criteria
- Priority
- Sprint fit
- Known assumptions or dependencies

## Definition of Done

A backlog item is done when:

- The expected output is complete
- Acceptance criteria are satisfied
- Jira status is updated
- Relevant evidence is available
- Human review is completed for AI-generated content
