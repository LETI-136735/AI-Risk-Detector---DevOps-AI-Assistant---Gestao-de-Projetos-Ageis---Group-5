# Sprint 1 Retrospective

## 1. Sprint Retrospective Purpose

The purpose of the Sprint 1 Retrospective is to inspect how the team worked during the first sprint and identify improvement actions for the next sprint.

The retrospective focuses on the process, collaboration, documentation, Scrum application and alignment with the project objectives.

Sprint 1 was mainly focused on defining the product idea, clarifying the MVP and creating the initial project structure.

---

## 2. What Went Well

| Area | Observation |
|---|---|
| Product idea | The team defined a clear and realistic product concept. |
| MVP scope | The team understood that the MVP should remain lightweight. |
| Agile alignment | The project was connected to backlog quality, feedback and incremental delivery. |
| Scrum structure | Initial Scrum roles, artifacts and events were identified. |
| Product Backlog | The team created a first version of the backlog with user stories and priorities. |
| DevOps connection | The product was linked to risk, quality, testing and release readiness. |
| Repository organisation | The repository started to become more structured and easier to evaluate. |

---

## 3. What Could Be Improved

| Area | Improvement Need |
|---|---|
| Acceptance criteria | Some user stories needed clearer Given/When/Then acceptance criteria. |
| Jira evidence | Jira needed to be aligned with the Product Backlog in GitHub. |
| Documentation structure | The README was too detailed and needed to become more introductory. |
| DevOps evidence | The DevOps approach needed to be explained more explicitly. |
| Responsible AI | The project needed a clearer statement about human validation of AI outputs. |
| Sprint evidence | Sprint Planning, Review and Retrospective evidence needed to be documented. |

---

## 4. Improvement Actions

The team identified the following improvement actions for Sprint 2.

| Action | Owner | Expected Result |
|---|---|---|
| Refine Product Backlog user stories | Team | Clearer stories with better acceptance criteria |
| Align Jira issues with GitHub backlog | Jira responsible/team | Consistent Scrum evidence |
| Improve README structure | Documentation responsible/team | Shorter and clearer repository introduction |
| Create DevOps approach document | Team | Stronger explanation of DevOps alignment |
| Add Responsible AI documentation | Team | Clearer human-in-the-loop principle |
| Prepare MVP demo evidence | Team | Better support for final evaluation |
| Draft final report structure | Team | Easier final submission preparation |

---

## 5. Link with Agile Continuous Improvement

The Sprint 1 Retrospective reflects the Agile principle of continuous improvement.

Instead of treating the first version of the project as final, the team used Sprint 1 to inspect both the product and the way of working.

The main learning was that the project should not focus only on the application. It should clearly demonstrate the complete Agile Project Management approach:

- product vision;
- prioritised backlog;
- Scrum execution;
- Jira evidence;
- DevOps reasoning;
- MVP demonstration;
- responsible AI reflection;
- final critical analysis.

---

## 6. Retrospective Summary

Sprint 1 helped the team clarify the direction of the project.

The team learned that a successful submission requires coherence between the product idea, Scrum evidence, DevOps explanation and final reflection.

The main improvement for Sprint 2 is to strengthen the documentation and evidence, while keeping the MVP simple and focused.
